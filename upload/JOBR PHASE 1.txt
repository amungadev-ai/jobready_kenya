# 🧠 Candidate-to-Job Matching Engine Documentation

## Version: Phase 2 Technical Architecture (v2.0 – Comprehensive)

---

## Core Principle

The platform uses the following principle:

**AI-on-Ingest, CPU-on-Match**

This means:

- AI or parser logic is used when a CV or job is uploaded.
- The uploaded CV or job post is converted into structured data.
- Structured data is saved in MySQL.
- Matching is done using SQL joins and scoring logic.
- AI is not used repeatedly during matching.
- CV files are not permanently stored.

This keeps the platform scalable, cheap, and fast.

---

## 1. Product Vision

The platform helps candidates discover jobs that match their real skills, qualifications, experience, and career interests.

The goal is not to match only job titles.

The goal is to understand:

- What the candidate can do
- What the job requires
- What category of work the role belongs to
- What subcategory or specialization the role belongs to
- What skills are required
- What qualifications are required
- What the candidate is interested in
- What jobs are realistic for the candidate

---

## 2. Matching Philosophy

The system separates:

- Organization type
- Organization industry
- Job category
- Job subcategory
- Role title
- Skills
- Tools
- Qualifications
- Experience
- Candidate preferences

### Example

Kenyatta National Hospital is in the healthcare industry.

But if they hire an HR Officer:

- Organization industry: Healthcare
- Job category: Human Resources
- Job subcategory: HR Generalist or Recruitment
- Role: HR Officer

The candidate does not need to be a healthcare worker.

They need to be an HR professional.

Therefore:

Industry is context.

Category and subcategory are strong matching signals.

Skills and qualifications are proof.

---

## 3. Why Industry Has Low Matching Weight

A professional can work across many industries.

Example:

An HR Officer can work in:

- Healthcare
- Banking
- Telecommunications
- Education
- Government
- Manufacturing
- NGO

A Finance Officer can work in:

- NGO
- Government
- Healthcare
- Banking
- Manufacturing
- Education

So industry should not dominate matching.

Recommended use:

- Industry familiarity is a small bonus.
- It should not reject otherwise good candidates.
- Category, subcategory, skills, and qualifications should carry more weight.

---

## 4. Final Matching Weights

Recommended scoring model:

```js
const MATCH_WEIGHTS = {
  categoryMatch: 25,
  subcategoryMatch: 25,
  skillsMatch: 25,
  qualificationsMatch: 10,
  experienceMatch: 10,
  industryFamiliarity: 5
};
```

Total: 100%

Meaning:

- Category match checks if the candidate is in the same career function.
- Subcategory match checks specialization.
- Skills match checks capability.
- Qualifications match checks eligibility.
- Experience match checks level.
- Industry familiarity gives a small contextual bonus.

## 5. System Architecture

### Candidate Upload Flow

```txt
Candidate uploads CV
        ↓
Backend receives file in memory
        ↓
Markitdown extracts Markdown/text
        ↓
Parser extracts structured data
        ↓
Optional AI fallback if parser confidence is low
        ↓
Normalize data against taxonomy
        ↓
Save structured profile to MySQL
        ↓
Discard uploaded file
        ↓
Run CPU-based matching
        ↓
Save top matches
        ↓
Return candidate profile + matches
```

---

### Job Upload Flow

```txt
Recruiter uploads job post
        ↓
Backend receives job text
        ↓
Parser or AI extracts structured job profile
        ↓
Normalize against taxonomy
        ↓
Save job and job profile to MySQL
        ↓
Run CPU-based matching against candidates
        ↓
Save candidate-job scores
```

---

## 6. No Media Storage Rule

CV files should not be permanently stored.

The file should be:

1. Uploaded.
2. Processed in memory.
3. Converted to text.
4. Discarded.

The database stores:

- Extracted name
- Email
- Phone
- Skills
- Education
- Work experience
- Qualifications
- Career category
- Subcategories
- Matching data

The database should not store the original CV file unless a future business requirement requires it.

Optional:

You may store raw extracted CV text for debugging for a short period, for example 30 days.

## 7. AI Role in the System

AI should not be the main matcher.

AI is used for:

### 7.1 CV Extraction

AI helps convert messy CVs into structured data.

Example:

```txt
Finance Intern at PATH Kenya
Managed Accounts Payable, reconciliations, audit schedules...
```

Becomes:

```json
{
  "category": "FINANCE_ACCOUNTING",
  "subcategories": ["ACCOUNTING", "ACCOUNTS_PAYABLE", "AUDIT"],
  "skills": ["ACCOUNTS_PAYABLE", "BANK_RECONCILIATION", "AUDIT_SUPPORT"],
  "tools": ["ERP_SYSTEMS", "EXCEL"],
  "industriesWorkedIn": ["INTERNATIONAL_DEVELOPMENT", "NON_PROFIT"]
}
```

### 7.2 Job Extraction

AI helps extract structured job requirements.

Example:

```txt
Part-Time Lecturer required for Community Health, Counselling Psychology, Social Work...
```

Becomes:

```json
{
  "jobCategory": "EDUCATION",
  "jobSubcategories": ["LECTURING", "TVET_TRAINING"],
  "specializations": [
    "COMMUNITY_HEALTH",
    "COUNSELLING_PSYCHOLOGY",
    "SOCIAL_WORK"
  ]
}
```

### 7.3 Normalization

AI or parser logic maps different wording to the same taxonomy item.

Examples:

```txt
Talent Acquisition → Recruitment
People Operations → HR Operations
Quick Books → QuickBooks
Accounts Clerk → Accounting
IT Helpdesk → IT Support
```

### 7.4 Transferable Skills Detection

AI helps detect career transitions.

Examples:

```txt
Cashier + CPA + reconciliation → Accounts Assistant
Radio Presenter + Social Media → Communications Officer
Enumerator + Data Collection → M&E Assistant
```

### 7.5 Explanation

After the algorithm calculates the score, AI can explain the match.

Example:

```txt
You are a strong match because your CV shows Accounts Payable, Bank Reconciliation, CPA progression, and financial reporting experience.

Your main gap is SAP experience, which was not found in your CV.
```

## 9. Taxonomy Model

The taxonomy is the foundation of the matching engine.

It stores:

- Organization types
- Industries
- Job categories
- Subcategories
- Roles
- Skills
- Tools
- Qualifications
- Certifications
- Specializations

### 9.1 taxonomy_items

```sql
CREATE TABLE taxonomy_items (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,

  type ENUM(
    'ORGANIZATION_TYPE',
    'INDUSTRY',
    'CATEGORY',
    'SUBCATEGORY',
    'ROLE',
    'SKILL',
    'TOOL',
    'QUALIFICATION',
    'CERTIFICATION',
    'SPECIALIZATION',
    'REGULATION',
    'COMPETENCY'
  ) NOT NULL,

  value VARCHAR(120) NOT NULL,
  label VARCHAR(255) NOT NULL,

  parent_id BIGINT UNSIGNED NULL,

  description TEXT NULL,

  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE KEY unique_taxonomy_value (type, value),

  INDEX idx_taxonomy_type (type),
  INDEX idx_taxonomy_parent (parent_id),

  CONSTRAINT fk_taxonomy_parent
    FOREIGN KEY (parent_id)
    REFERENCES taxonomy_items(id)
    ON DELETE SET NULL
);
```

Example rows:

```txt
CATEGORY: HUMAN_RESOURCES
SUBCATEGORY: RECRUITMENT
SKILL: CANDIDATE_SCREENING
TOOL: QUICKBOOKS
QUALIFICATION: BACHELOR_DEGREE
INDUSTRY: HEALTHCARE
```

### 9.3 taxonomy_relationships

This supports adjacent career relationships.

Example:

```txt
Finance is related to Banking
Finance is related to Insurance
```

```sql
CREATE TABLE taxonomy_relationships (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,

  source_item_id BIGINT UNSIGNED NOT NULL,
  target_item_id BIGINT UNSIGNED NOT NULL,

  relationship_type ENUM(
    'RELATED',
    'ADJACENT',
    'TRANSFERABLE',
    'PARENT_CHILD',
    'SIMILAR'
  ) NOT NULL DEFAULT 'RELATED',

  weight DECIMAL(4,2) NOT NULL DEFAULT 1.00,

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  UNIQUE KEY unique_relationship (source_item_id, target_item_id, relationship_type),

  CONSTRAINT fk_relationship_source
    FOREIGN KEY (source_item_id)
    REFERENCES taxonomy_items(id)
    ON DELETE CASCADE,

  CONSTRAINT fk_relationship_target
    FOREIGN KEY (target_item_id)
    REFERENCES taxonomy_items(id)
    ON DELETE CASCADE
);
```

## 18. Match Calculation Logic

### 18.1 Category Match

```js
function calculateCategoryMatch(candidate, job) {
  if (candidate.primaryCategoryId === job.jobCategoryId) {
    return 100;
  }

  if (candidate.interestCategoryIds.includes(job.jobCategoryId)) {
    return 75;
  }

  return 0;
}
```

### 18.2 Subcategory Match (✅ UPDATED)

```js
function calculateSubcategoryMatch(candidate, job) {
  // Primary subcategory match = highest score
  if (candidate.primarySubcategoryId === job.primarySubcategoryId) {
    return 100;
  }

  // Check candidate's extracted subcategories against job's subcategories
  const candidateSubcategories = candidate.subcategoryIds;
  const jobSubcategories = job.subcategoryIds;

  if (!jobSubcategories.length) return 0;

  const matched = jobSubcategories.filter(id =>
    candidateSubcategories.includes(id)
  );

  const score = (matched.length / jobSubcategories.length) * 100;

  // If candidate has it as an interest, boost slightly
  if (candidate.interestCategoryIds.includes(job.jobCategoryId)) {
    return Math.min(score + 10, 100);
  }

  return score;
}
```

### 18.3 Skills Match

```js
function calculateSkillsMatch(candidateSkillIds, jobRequiredSkillIds) {
  if (!jobRequiredSkillIds.length) return 100;

  const matched = jobRequiredSkillIds.filter(id =>
    candidateSkillIds.includes(id)

}
```

```js
function calculateExperienceMatch(candidateYears, minimumYears) {

  if (candidateYears >= minimumYears) return 100;

  return (candidateYears / minimumYears) * 100;
```

## 19. SQL-Based Matching Query (✅ UPDATED)

This is the CPU-based matching approach.

Important:

Use taxonomy IDs, not raw skill names.

### Candidate to Jobs Matching

```sql
SELECT
  j.id AS job_id,
  j.title,
  j.location_county,
  j.is_remote,
  c.name AS company_name,

  -- Category Score (Weight: 25%)
  CASE
    WHEN cp.primary_category_id = jp.job_category_id THEN 100
    WHEN EXISTS (
      SELECT 1
      FROM candidate_interests ci
      WHERE ci.candidate_id = ca.id
        AND ci.category_id = jp.job_category_id
    ) THEN 75
    ELSE 0
  END AS category_score,

  -- ✅ UPDATED: Subcategory Score (Weight: 25%)
  CASE
    -- Highest: Primary subcategory matches
    WHEN jp.primary_subcategory_id IS NOT NULL
      AND cp.primary_subcategory_id = jp.primary_subcategory_id
    THEN 100

    -- High: Candidate's extracted subcategories match job's subcategories
    WHEN EXISTS (
      SELECT 1
      FROM job_subcategories js
      INNER JOIN candidate_subcategories cs
        ON cs.candidate_id = ca.id
        AND cs.subcategory_id = js.subcategory_id
      WHERE js.job_id = j.id
    ) THEN 70

    -- Medium: Candidate has the job's category as an interest
    WHEN EXISTS (
      SELECT 1
      FROM candidate_interests ci
      WHERE ci.candidate_id = ca.id
        AND ci.category_id = jp.job_category_id
    ) THEN 50

    ELSE 0
  END AS subcategory_score,

  -- Skills Score (Weight: 25%)
  (
    SELECT
      COALESCE(
        (COUNT(DISTINCT csk.skill_id) / NULLIF(COUNT(DISTINCT jsk.skill_id), 0)) * 100,
        100
      )
    FROM job_skills jsk
    LEFT JOIN candidate_skills csk
      ON csk.candidate_id = ca.id
     AND csk.skill_id = jsk.skill_id
    WHERE jsk.job_id = j.id
      AND jsk.importance = 'MUST_HAVE'
  ) AS skills_score,

  -- Qualifications Score (Weight: 10%)
  (
    SELECT
      COALESCE(
        (COUNT(DISTINCT cq.qualification_id) / NULLIF(COUNT(DISTINCT jq.qualification_id), 0)) * 100,
        100
      )
    FROM job_qualifications jq
    LEFT JOIN candidate_qualifications cq
      ON cq.candidate_id = ca.id
     AND cq.qualification_id = jq.qualification_id
    WHERE jq.job_id = j.id
      AND jq.importance = 'MUST_HAVE'
  ) AS qualifications_score,

  -- Experience Score (Weight: 10%)
  CASE
    WHEN jp.minimum_experience_years IS NULL OR jp.minimum_experience_years = 0 THEN 100
    WHEN cp.total_experience_years >= jp.minimum_experience_years THEN 100
    ELSE (cp.total_experience_years / jp.minimum_experience_years) * 100
  END AS experience_score,

  -- Industry Score (Weight: 5%)
  CASE
    WHEN jp.organization_industry_id IS NULL THEN 50
    WHEN EXISTS (
      SELECT 1
      FROM candidate_work_experiences cwe
      WHERE cwe.candidate_id = ca.id
        AND cwe.organization_industry_id = jp.organization_industry_id
    ) THEN 100
    ELSE 30
  END AS industry_score,

  -- ✅ OPTIMIZATION: Count matched skills for UI
  (
    SELECT COUNT(DISTINCT csk.skill_id)
    FROM job_skills jsk
    INNER JOIN candidate_skills csk
      ON csk.candidate_id = ca.id
      AND csk.skill_id = jsk.skill_id
    WHERE jsk.job_id = j.id
      AND jsk.importance = 'MUST_HAVE'
  ) AS matched_skill_count,

  (
    SELECT COUNT(DISTINCT jsk.skill_id)
    FROM job_skills jsk
    WHERE jsk.job_id = j.id
      AND jsk.importance = 'MUST_HAVE'
  ) AS total_required_skills

FROM candidates ca
JOIN candidate_profiles cp
  ON cp.candidate_id = ca.id

JOIN jobs j
  ON j.status = 'ACTIVE'
 AND j.deleted_at IS NULL

JOIN companies c
  ON c.id = j.company_id

JOIN job_profiles jp
  ON jp.job_id = j.id

WHERE ca.id = ?
  AND ca.onboarding_status = 'COMPLETED'
  AND j.extraction_status IN ('EXTRACTED', 'NEEDS_REVIEW')
  AND j.application_deadline >= CURDATE()
```

### Application Logic

After fetching the SQL results:

```js
// Calculate final score using the weights
const finalScore = (
  (row.category_score * 0.25) +
  (row.subcategory_score * 0.25) +
  (row.skills_score * 0.25) +
  (row.qualifications_score * 0.10) +
  (row.experience_score * 0.10) +
  (row.industry_score * 0.05)
);

// Determine verdict
const verdict = getVerdict(finalScore);

// Determine recommendation type
const recommendationType = 
  row.subcategory_score >= 80 ? 'PRIMARY_MATCH' :
  row.category_score >= 75 ? 'INTEREST_MATCH' :
  row.skills_score >= 60 ? 'ADJACENT_MATCH' :
  row.skills_score >= 40 ? 'STRETCH_MATCH' :
  'NOT_RECOMMENDED';
```

## 21. Background Cron

Shared-hosting cron is used instead of Redis, BullMQ, or queue workers.

### Cron endpoint

```txt
POST /api/cron/match?token=SECRET
```

### cPanel command

```bash
curl -X POST https://yourdomain.com/api/cron/match?token=YOUR_SECRET
```

Run every 5–10 minutes.

---

### Cron Responsibilities

The cron should:

1. Find newly uploaded candidates.
2. Find newly posted jobs.
3. Match new candidates against active jobs.
4. Match new jobs against active candidates.
5. Save scores in candidate_job_scores.
6. Update system_settings.LAST_CRON_MATCH_RUN.

---

### Example Node.js Cron Endpoint

```js
app.post("/api/cron/match", async (req, res) => {
  const token = req.query.token;

  if (token !== process.env.CRON_SECRET) {
    return res.status(401).json({ error: "Unauthorized" });
  }

  const lastRun = await getSystemSetting("LAST_CRON_MATCH_RUN");

  // ✅ UPDATED: Only fetch candidates with COMPLETED onboarding
  const newCandidates = await db.query(`
    SELECT id
    FROM candidates
    WHERE created_at >= ?
      AND onboarding_status = 'COMPLETED'
  `, [lastRun]);

  // ✅ UPDATED: Only fetch ACTIVE jobs with EXTRACTED status
  const newJobs = await db.query(`
    SELECT id
    FROM jobs
    WHERE created_at >= ?
      AND status = 'ACTIVE'
      AND extraction_status IN ('EXTRACTED', 'NEEDS_REVIEW')
      AND deleted_at IS NULL
      AND (application_deadline IS NULL OR application_deadline >= CURDATE())
  `, [lastRun]);

  for (const candidate of newCandidates) {
    await computeMatchesForCandidate(candidate.id);
  }

  for (const job of newJobs) {
    await computeMatchesForJob(job.id);
  }

  await setSystemSetting("LAST_CRON_MATCH_RUN", new Date().toISOString());

  return res.json({
    success: true,
    processedCandidates: newCandidates.length,
    processedJobs: newJobs.length
  });
});
```

## 27. Performance Strategy

For 5,000 jobs and 2,000 candidates per day:

Do not run AI comparisons.

Bad approach:

```txt
5,000 jobs × 2,000 candidates = 10,000,000 AI comparisons
```

Good approach:

```txt
2,000 CV extractions
+ new job extractions
+ SQL matching
```

Matching runs with:

- Indexed joins
- Cached candidate_job_scores
- Cron batch processing
- Optional synchronous first 100 matches after upload
